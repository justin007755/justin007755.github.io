Rsync研究

Mar 31, 2016 | IT技术 | Hits

Rsync是Linux里一个常用的工具,可以用来在不同的机器之间同步文件.相对于其它工具(比如SCP)而言它的效率比较高,因为rsync的算法是只拷贝哪些改动了的数据,而SCP是全盘复制.对于rsync的同步算法本人没有详细了解,在网上有不少材料可以自行学习.

本人对rsync的研究源于工作中的一个真实案例.我们在某客户处部署了一套网络设备,另外还有一套相应的网管设备.根据我们的设计,网络设备将用rsync工具向网管设备定期同步数据.某一天网管设备出了些问题,我们把它重启了,然后我们发现网管设备上的rsync配置文件不见了,而且rsync daemon用”ps -ef | grep rsync”也查不到了,但奇怪的是两台设备之间的数据同步并未受影响,百思不得其姐啊…:-)

后来咨询了公司里比较资深的专家,也查了不少文档,终于发现rsync其实是有两种工作模式.一种是命令行模式,另一种是服务模式.其实这两种模式在rsync的menu(“man rsync”: Rsync is a fast and extraordinarily versatile file copying tool. It can copy locally, to/from another host over any remote shell, or to/from a remote rsync daemon.)里都有说明,没仔细看啊(充分说明所有问题都可以从源头找到答案).命令行模式其实就是采用SSH用push或者pull的方式来同步文件,换句话来说就是rsync的客户端通过ssh激活了目的主机的rsync server,一旦同步结束,该rsync server也就停了.这就是为什么我们之前的问题会发生(用ps查不到rsync的进程).

一个有意思的情况是我即使明白了这个道理,一开始我还是在目的主机上抓不到rsync server的那个sshd的连接,后来发现这个连接结束的太快,而且目的主机的NTP未同步,所以写了下面的脚本并同步了NTP才最终抓到了想要的ssh连接(用netstat或者ss).
for ((i=1; i<2000; i="i+1))" ;="" do="" (netstat="" -tpn="" |="" grep="" ssh="" &&="" date="" echo="" "="===========")">> /tmp/test ; done

关于rsync的服务模式,鸟哥的文章介绍的比较详细,参考即可,不赘述.

源主机,发起rysnc连接.

目的主机,由源主机发起的rsync激活rsync server(10484/sshd),同步结束后该rsync server也随之结束.

NAME
rsync — a fast, versatile, remote (and local) file-copying tool

SYNOPSIS
Local: rsync [OPTION…] SRC… [DEST]

Access via remote shell:
  Pull: rsync [OPTION...] [USER@]HOST:SRC... [DEST]
  Push: rsync [OPTION...] SRC... [USER@]HOST:DEST

Access via rsync daemon:
  Pull: rsync [OPTION...] [USER@]HOST::SRC... [DEST]
        rsync [OPTION...] rsync://[USER@]HOST[:PORT]/SRC... [DEST]
  Push: rsync [OPTION...] SRC... [USER@]HOST::DEST
        rsync [OPTION...] SRC... rsync://[USER@]HOST[:PORT]/DEST

USING RSYNC-DAEMON FEATURES VIA A REMOTE-SHELL CONNECTION
It is sometimes useful to use various features of an rsync daemon (such as named modules) without actually
allowing any new socket connections into a system (other than what is already required to allow remote-shell
access). Rsync supports connecting to a host using a remote shell and then spawning a single-use “daemon”
server that expects to read its config file in the home dir of the remote user. This can be useful if you want
to encrypt a daemon-style transfer’s data, but since the daemon is started up fresh by the remote user, you may
not be able to use features such as chroot or change the uid used by the daemon. (For another way to encrypt a
daemon transfer, consider using ssh to tunnel a local port to a remote machine and configure a normal rsync
daemon on that remote host to only allow connections from “localhost”.)
